# Plausible Analytics - Kamal Deployment

Self-hosted [Plausible Analytics](https://plausible.io/) Community Edition deployed with [Kamal 2](https://kamal-deploy.org/).

Designed for shared servers -- kamal-proxy routes traffic by hostname, so other Kamal-deployed apps can coexist on the same machine.

## Prerequisites

- [Kamal 2](https://kamal-deploy.org/) (`gem install kamal`)
- [just](https://github.com/casey/just) command runner
- A server with SSH root access (Ubuntu/Debian recommended)
- A Docker registry account ([Docker Hub](https://hub.docker.com/), [GitHub Container Registry](https://ghcr.io), etc.)
- A domain name with DNS pointed at your server

## Getting Started

1. Clone and initialize secrets:

   ```bash
   git clone <this-repo> && cd kamal-plausible
   just init
   ```

   This creates `.kamal/secrets` with auto-generated values for `SECRET_KEY_BASE`, `TOTP_VAULT_KEY`, and `POSTGRES_PASSWORD`.

2. Edit `.kamal/secrets` and fill in the four remaining values:

   ```
   SERVER_IP=203.0.113.1
   PLAUSIBLE_HOST=analytics.example.com
   REGISTRY_USERNAME=yourdockerhubuser
   KAMAL_REGISTRY_PASSWORD=dckr_pat_xxxxx
   ```

3. Deploy:

   ```bash
   just setup
   ```

4. Visit `https://analytics.example.com/register` to create your admin account. Plausible allows one signup even with invite-only registration when no users exist yet.

## Commands

| Command | Description |
|---|---|
| `just init` | Create `.kamal/secrets` with generated cryptographic values |
| `just setup` | First-time server provisioning and deploy |
| `just deploy` | Deploy or update Plausible |
| `just logs` | Follow application logs |
| `just logs-db` | Follow PostgreSQL logs |
| `just logs-events` | Follow ClickHouse logs |
| `just reboot-db` | Reboot PostgreSQL |
| `just reboot-events` | Reboot ClickHouse |

## Updating Plausible

Change the version tag in `Dockerfile` and redeploy:

```dockerfile
FROM ghcr.io/plausible/community-edition:v3.3.0
```

```bash
just deploy
```

## Configuration

All configuration lives in `.kamal/secrets` (gitignored). The `deploy.yml` uses ERB templates to read values from it, so you never need to edit `deploy.yml` for a standard deployment.

### Required

| Variable | Description |
|---|---|
| `SERVER_IP` | Server IP address |
| `PLAUSIBLE_HOST` | Domain name (e.g. `analytics.example.com`) |
| `REGISTRY_USERNAME` | Docker registry username |
| `KAMAL_REGISTRY_PASSWORD` | Docker registry password or access token |
| `SECRET_KEY_BASE` | App secret (auto-generated by `just init`) |
| `TOTP_VAULT_KEY` | 2FA encryption key (auto-generated by `just init`) |
| `POSTGRES_PASSWORD` | Database password (auto-generated by `just init`) |

### Optional: Email

Needed for user invitations and password resets. Uncomment in `.kamal/secrets`:

```
SMTP_HOST_ADDR=smtp.example.com
SMTP_HOST_PORT=587
SMTP_HOST_SSL_ENABLED=true
SMTP_USER_NAME=apikey
SMTP_USER_PWD=your-smtp-password
MAILER_EMAIL=plausible@example.com
MAILER_NAME=Plausible Analytics
```

### Optional: MaxMind GeoIP

For city-level visitor geolocation (by default, Plausible uses a country-level database):

```
MAXMIND_LICENSE_KEY=your-license-key
MAXMIND_EDITION=GeoLite2-City
```

### Optional: Google Search Console

```
GOOGLE_CLIENT_ID=your-client-id
GOOGLE_CLIENT_SECRET=your-client-secret
```

## Architecture

```
Internet
  |
  v
kamal-proxy (ports 80/443, auto SSL via Let's Encrypt)
  |
  v  (Docker network: kamal)
plausible (port 8000) --> plausible-db (PostgreSQL 16)
                      --> plausible-events-db (ClickHouse 24.12)
```

- **kamal-proxy** terminates TLS and routes by hostname. Other apps on the same server use different `proxy.host` values in their own Kamal configs.
- **Databases** have no ports published to the host -- only reachable via Docker's internal network.
- **ClickHouse** runs in low-resource mode (single-threaded, reduced memory). Remove `clickhouse/default-profile-low-resources-overrides.xml` and `clickhouse/low-resources.xml` if you have a dedicated server.
- On every deploy, migrations run automatically before the app starts.

## Shared Server

This setup coexists with other Kamal deployments on the same server. Each app gets routed by its `proxy.host` value. No additional configuration is needed -- just deploy your other apps to the same `SERVER_IP` with their own domain names.
